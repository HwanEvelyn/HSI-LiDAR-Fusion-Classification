# v2阶段总结与系统验收清单

## 1. 文档目的

这份文档主要用于整理 HCT-BGC v2 阶段在 Houston 数据集上的实验进展，形成一份比较完整的阶段总结，同时列出后续做系统验收时需要准备的成果物清单。这样做的目的，一方面是把目前已经做过的尝试和结论整理清楚，避免后面写论文、写阶段报告或者准备答辩材料时再零散回忆；另一方面也是为了给后续的系统验收、材料归档和结果复现实验提供一个统一的参照。

从目前的实验情况来看，v2 已经完成了从基础方案到多尺度方案、从简单融合到残差融合、从无辅助头到有辅助头、再到蒸馏尝试的一轮较系统的探索。整体上，v2 的设计方向是有效的，而且已经得到了比较明确的阶段性最优配置。

---

## 2. v2阶段的目标和整体思路

### 2.1 阶段目标

v2 阶段的目标，不只是简单把指标做高，而是希望在 v1 的基础上进一步回答下面几个问题：

- 融合模块是否还能继续优化，尤其是不同尺度信息应该怎样进入主干表示。
- 在 Houston 这种类别多、场景复杂、异质性明显的数据集上，多尺度上下文是否真的有帮助。
- 辅助监督是否能够帮助主分支训练得更稳定、更充分。
- 在已有结构基础上加入蒸馏，是否能够继续提高泛化性能。
- 如果要把 v2 作为后续论文和系统验收的主要版本，哪一个配置最适合作为最终交付方案。

### 2.2 整体改进思路

从已有配置文件和实验记录来看，v2 主要围绕下面几个方向展开：

- 编码器保持 `light_hetero` 这一相对轻量的异构建模思路。
- 保留 `conservative fusion`，避免过强融合导致特征扰动过大。
- 在尺度融合方式上，从 `average` 过渡到 `residual`。
- 在空间建模上，从单尺度 `patch_size = 11` 扩展到 `patch_size = 11, context_patch_size = 17` 的多尺度输入。
- 引入辅助头 `aux heads`，并比较 `linear` 和 `mlp` 两种形式。
- 在辅助头基础上进一步尝试 `distill`，验证蒸馏是否有效。

可以认为，v2 的主线不是“盲目加模块”，而是围绕“怎样更稳、更合理地融合局部与上下文信息”来做结构优化。

## 3. v2实验配置路线梳理

### 3.1 v2.0 基线方案

当前仓库中的 `experiments/hct_bgc_v2.0_houston.yaml` 对应的是 v2.0 阶段的基础配置，其主要特征如下：

- `patch_size = 11`
- `scale_fusion_mode = average`
- `encoder_variant = light_hetero`
- `use_conservative_fusion = true`
- `use_aux_heads = false`

这一版可以理解为 v2 的起点版本。它已经不是最初的 baseline，而是在 v1 之后进一步整理过的一版较稳妥结构，用来作为后续所有 v2 变体的比较基准。

### 3.2 residual + aux 方案

在 v2.0 的基础上，后续尝试将 `scale_fusion_mode` 从 `average` 改为 `residual`，并引入辅助头监督。这个改动的核心目的，是让尺度融合由“直接平均”变成“主干特征 + 上下文残差补充”的方式，从而减少不同尺度特征在语义上相互冲突的问题。

这一阶段的实验结果说明，`residual` 方案比 `average` 更有效，而且效果比较稳定，是 v2 中最关键的一次结构改进。

### 3.3 多尺度 11×17 上下文方案

当前仓库中的 `experiments/hct_bgc_v2_houston_multiscale_11_17.yaml` 对应的是多尺度方案，其核心配置为：

- `patch_size = 11`
- `context_patch_size = 17`
- `context_token_size = 11`
- `scale_fusion_mode = residual`
- `encoder_variant = light_hetero`
- `use_conservative_fusion = true`
- `use_aux_heads = true`

这版实验的重点，是在保持中心 patch 为 11 的同时，引入更大范围的 17×17 上下文窗口，再通过 `context_token_size = 11` 控制上下文信息进入模型时的表达粒度。这个设计本质上是在做“局部精细判别 + 外围上下文补充”的结合。

### 3.4 aux head 形式比较

在多尺度方案上，还继续比较了 `aux_head_mode = linear` 和 `aux_head_mode = mlp` 两种辅助头形式。

- `aux = linear` 的特点是结构简单、额外参数少、训练目标更直接。
- `aux = mlp` 的特点是表达能力更强，但也可能更容易引入额外波动或者过拟合。

从当前结果看，`linear` 更适合作为最终正式方案。

### 3.5 distill 尝试

仓库中的 `experiments/hct_bgc_v2_1_ms_houston.yaml` 是在多尺度方案基础上继续加入蒸馏的尝试，主要增加了：

- `aux_head_mode = linear`
- `aux_distill_weight = 0.1`
- `aux_distill_temperature = 2.0`

这一版的目的，是希望通过主输出对辅助头进行蒸馏约束，进一步提升辅助分支与主分支的一致性。不过从结果来看，蒸馏没有表现出稳定正收益。

### 4.2 当前最优结论

从多次重复实验的均值结果来看，当前 v2 阶段综合性能最好的方案是：

`Houston（patch 11 × 17）+ residual + context-token-size=11 + aux=linear`

对应测试集指标为：

- `Test OA = 0.8685 ± 0.0104`
- `Test AA = 0.8817 ± 0.0055`
- `Test Kappa = 0.8576 ± 0.0111`

这说明当前最值得作为 v2 正式代表配置的，不是单次成绩看起来更高的某个偶然结果，而是**经过重复实验后综合指标最高、结论最稳的一版**。

## 5. v2阶段的重点分析

这一部分是 v2 总结中最重要的内容。后续无论是写阶段汇报、论文实验分析，还是准备系统验收材料，这部分都可以直接作为主体。

### 5.1 从 average 到 residual：说明尺度融合方式确实重要

v2.0 使用的是 `scale_fusion_mode = average`。从结果上看，这一版虽然已经具备一定性能，但测试集 `OA = 0.8432`，`Kappa = 0.8304`，和后续方案相比仍有明显差距。

当把尺度融合方式改成 `residual` 并加入辅助头后，测试集指标提升到了：

- `Test OA = 0.8638`
- `Test Kappa = 0.8526`

与 v2.0 相比：

- `Test OA` 提升约 `2.06` 个百分点
- `Test Kappa` 提升约 `2.22` 个百分点

这个结果说明，在多尺度或多来源特征融合时，简单平均会削弱主干特征的判别性，而残差式融合更容易保留主分支的稳定语义，同时把新增的上下文信息作为增量补充进去。对于 Houston 这种类别细粒度差异比较明显的数据集，这种融合方式显然更合适。

换句话说，v2 的第一个关键结论是：**比起继续堆更复杂的模块，先把“信息怎么融合”这件事做对，收益更大。**

### 5.2 辅助头是有效的，但更适合用轻量方式实现

引入辅助头以后，模型训练过程中不仅优化主分类输出，也同时对两个辅助分支进行约束，这样做通常有两个潜在好处：

- 可以让模态相关特征在中间层得到更明确的监督。
- 可以在训练前期提供更充足的梯度信号，提升优化稳定性。

从结果上看，辅助头的引入和 residual 融合配合较好，至少在当前实验中，它没有拖后腿，反而帮助模型进一步提升了性能。

不过，当比较 `aux = linear` 和 `aux = mlp` 时，当前证据更支持 `linear`：

- `aux = linear` 有多次重复实验支撑，结果可靠。
- `aux = mlp` 目前只有单次结果，虽然验证集指标更高，但测试集并没有超过 `aux = linear` 的重复均值结果。

因此，这里更合理的结论不是“MLP 不好”，而是：**现阶段没有足够证据证明 MLP 辅助头优于线性辅助头，所以正式方案应优先选择更简单、可复现性更好的 `linear`。**

### 5.3 多尺度 11×17 上下文是当前 v2 的核心增益来源之一

在 residual + aux 的基础上，再引入 `patch 11 × 17` 多尺度上下文后，测试集指标继续上升，达到当前最优的：

- `Test OA = 0.8685 ± 0.0104`
- `Test Kappa = 0.8576 ± 0.0111`

相对 v2.0，增益为：

- `Test OA` 提升约 `2.53` 个百分点
- `Test Kappa` 提升约 `2.72` 个百分点

这说明在 Houston 数据集上，仅靠中心 11×11 patch 仍然不能完全覆盖目标判别所需的上下文信息，而引入更大范围的 17×17 视野后，模型可以更充分利用周围空间结构、邻域语义和模态补充信息。

不过也要注意到，多尺度方案的收益并不是“所有指标一起大幅上涨”。例如：

- `Test OA` 和 `Test Kappa` 达到最优。
- `Test AA` 没有明显超过前一版 residual + aux=`linear`。

这说明多尺度更可能提升的是整体分类一致性和主体类别判别能力，而对所有类别的均衡改善并不是同样明显。这个现象后续可以结合 per-class accuracy 再做更细分析。

### 5.4 验证集更高，不代表测试集一定更强

`11×17 + aux=mlp` 的单次结果中，验证集指标看起来非常亮眼：

- `Val OA = 0.9476`
- `Val Kappa = 0.9426`

但其测试集结果：

- `Test OA = 0.8661`
- `Test Kappa = 0.8548`

并没有超过 `11×17 + aux=linear` 的重复实验均值结果。

这一点其实非常重要，因为它提醒我们：在当前阶段做方案选型时，不能只看某一次验证集最优，也不能只看一次运行结果。对于后续系统验收和正式汇报来说，**重复实验均值和标准差比单次最高值更有说服力**。从这个角度讲，`aux=linear` 更适合作为正式版本。

### 5.5 蒸馏目前没有形成稳定正收益

在 `11×17 + aux=linear` 的基础上加入蒸馏后，结果为：

- `Test OA = 0.8636 ± 0.0079`
- `Test Kappa = 0.8521 ± 0.0086`

相比未蒸馏版本：

- `Test OA` 略有下降
- `Test AA` 略有下降
- `Test Kappa` 略有下降

同时，验证集标准差也明显变大，说明蒸馏至少在当前参数设置下没有让训练变得更稳，反而可能增加了优化过程中的不确定性。

我认为这里可以得出一个比较稳妥的结论：**蒸馏不是当前 v2 的有效主增益项，至少暂时不适合写成核心贡献。**  
如果后面时间充足，它可以保留为后续优化方向；如果时间有限，它更适合归入“已尝试但未取得稳定提升”的探索项。

### 5.6 v2 当前已经形成比较明确的推荐配置

综合考虑性能、稳定性、结构复杂度和重复实验支撑情况，当前最适合作为 v2 正式推荐配置的是：

- `patch_size = 11`
- `context_patch_size = 17`
- `context_token_size = 11`
- `scale_fusion_mode = residual`
- `encoder_variant = light_hetero`
- `use_conservative_fusion = true`
- `use_aux_heads = true`
- `aux_head_mode = linear`

这版配置的优势主要有三点：

- 指标上是目前 v2 综合最优。
- 结论来自重复实验，而不是单次偶然结果。
- 结构改动虽然比 v2.0 多，但整体仍然保持在可以解释、可以复现、可以交付的范围内。

### 5.7 当前仍需要正视的局限性

虽然 v2 已经取得了阶段性最优结果，但也不能把结论写得过满。当前仍然有几个需要诚实说明的问题：

- 验证集和测试集之间仍然存在一定落差，说明泛化问题虽然有所改善，但没有被完全解决。
- `aux=mlp` 的结果还不够充分，暂时无法做出最终判断。
- 蒸馏尝试未取得稳定收益，说明当前辅助监督策略还没有完全挖掘干净。
- 目前的分析主要基于 OA、AA、Kappa 三项总体指标，后续仍需结合 confusion matrix 和 per-class accuracy 来进一步解释哪些类别真正受益。

因此，v2 当前更准确的表述应该是：**已经形成了稳定有效的改进方向和可交付的推荐版本，但仍有一些细节值得后续继续打磨。**

---

## 6. v2阶段总体结论

结合目前全部实验结果，我认为 v2 阶段可以概括为下面几条结论：

1. v2 的改进方向是有效的，尤其是 `residual` 融合和 `11×17` 多尺度上下文，为性能提升提供了主要贡献。
2. `light_hetero + conservative fusion` 这一整体框架是可以保留的，说明当前基础建模路线没有走偏。
3. 辅助头是值得保留的训练机制，但从已有证据看，正式方案优先采用 `linear`，不优先采用 `mlp`。
4. 蒸馏暂时没有形成稳定收益，不建议作为系统验收时的核心方案。
5. 当前最推荐作为 v2 正式版本的方案是：`11×17 multiscale + residual + context-token-size=11 + aux=linear`。

如果后面要把 v2 写进阶段报告或者论文实验总结里，那么最核心的一句话可以写成：

> v2 阶段通过将尺度融合方式由平均融合改为残差融合，并进一步引入 11×17 多尺度上下文与线性辅助头监督，在 Houston 数据集上取得了当前阶段最优且较稳定的综合性能，说明该版本已经具备作为正式交付方案的基础。

  重点分析：
  - `average` vs `residual`
  - single-scale vs multi-scale
  - `aux=linear` vs `aux=mlp`
  - with vs without distill

- 失败尝试记录  
  这一部分建议保留，不要删除。因为在系统验收或者答辩时，老师很可能会问“你还尝试过什么”。  
  可以明确记录：
  - 蒸馏为什么没有纳入最终方案
  - `aux=mlp` 为什么暂时不选
  - 哪些结果只有单次实验，证据不够充分

- 局限性分析  
  建议包括：
  - 泛化落差仍然存在
  - 某些类别的均衡提升可能有限
  - 多尺度引入后计算量会略有增加

## 7.6 模型交付类成果物

这一部分是“系统”验收里很容易被忽略，但实际很重要的一部分。

- 最终模型权重文件
- 最终配置文件
- 推理脚本
- 单张或批量推理入口
- 输入输出格式说明
- 结果保存路径说明

如果后面有演示环节，还建议额外准备：

- 一份最小可运行 demo
- 一份固定输入样例
- 一份对应的标准输出样例

## 7.7 文档与汇报类成果物

这一部分主要用于答辩、汇报和论文写作。

- 阶段总结文档
- 系统验收说明书
- PPT 或答辩展示材料
- 论文实验章节草稿
- 图表素材整理文件夹

建议把所有图表统一放到固定目录中，避免后面找不到：

- 主结果表
- 消融表
- confusion matrix 图
- 分类图
- loss 曲线图

## 8. 建议的系统验收材料组织方式

为了避免后面材料很乱，我觉得可以按下面的结构整理：

```text
thesis/
  notes/
  figures/
  tables/
  acceptance/
    configs/
    logs/
    checkpoints/
    reports/
    scripts/
```

其中：

- `notes/` 放阶段总结和实验记录
- `figures/` 放图片类材料
- `tables/` 放表格类材料
- `acceptance/configs/` 放最终验收配置
- `acceptance/logs/` 放关键训练日志
- `acceptance/checkpoints/` 放模型权重
- `acceptance/reports/` 放验收文档
- `acceptance/scripts/` 放复现和推理脚本

这样整理的好处是，后续无论是自己写论文，还是老师要检查材料，都会比较清楚。

## 9. 后续建议安排

为了顺利完成系统验收，后续建议按下面的顺序推进。

### 9.1 第一优先级：固定最终方案

先把最终方案确定为：

`11×17 multiscale + residual + context-token-size=11 + aux=linear`

然后围绕这版方案补齐所有需要的结果材料，避免后面在多个候选方案之间反复摇摆。

### 9.2 第二优先级：补齐最终方案的支撑材料

优先补齐：

- confusion matrix
- per-class accuracy
- classification map
- 多次重复实验结果表
- 与 v2.0 的增益对比表

因为这些内容最容易直接进入论文、汇报和验收材料。

### 9.3 第三优先级：规范成果物归档

建议尽快把下面几类东西固定下来：

- 最终权重命名方式
- 最终配置命名方式
- 结果目录命名方式
- 图表导出命名方式

如果这些不尽早统一，后期整理材料会比较痛苦。

### 9.4 第四优先级：处理可选优化项

如果时间还有余量，再考虑：

- 是否补 `aux=mlp` 的重复实验
- 是否重新调蒸馏参数
- 是否进一步分析 per-class 改善来源

这些工作不是当前系统验收的刚需，但可以作为论文增强项。

## 10. 最终结论

总体来说，v2 阶段已经完成了一轮比较完整的结构探索，并且形成了比较明确的结论。最关键的认识是：v2 的收益主要来自更合理的尺度融合方式和多尺度上下文建模，而不是单纯增加复杂度。当前最优方案 `11×17 multiscale + residual + aux=linear` 已经在多次重复实验下表现出最好的综合性能，因此完全可以作为后续系统验收和正式汇报的主版本。

接下来最重要的工作，不再是继续无休止地尝试新结构，而是围绕当前最优配置，把结果证明做扎实、把材料整理完整、把系统验收所需成果物补齐。只要后续把复现脚本、指标表、混淆矩阵、分类图、模型权重和验收文档这些内容组织好，v2 就已经具备进入系统验收阶段的基础。
